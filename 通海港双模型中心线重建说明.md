## 2026-05-18 方向纠偏

用户已明确：这段工作不是继续沿用 Hermes 之前的路线做局部修补，而是要推翻之前依赖旋转对齐 DOM 和旧中心线网络的思路，重新以原始生产 DOM 为主入口。

因此，本文件从现在开始定位为“历史实验与辅助证据记录”：

- 旋转对齐走廊 DOM 的训练结果只能说明模型在该分布上有效。
- `topology_skeleton_v1/v2` 可以作为通海港站场结构参考，但不是最终生产中心线。
- 旧 `final_centerline_network.shp` 和旧 refine/hybrid 结果不再作为可继续微调的主干。
- 手持 LAS 和点云单轨补全只作为轨距、Z、高度连续性和局部几何校验模块。
- 新的主线文档是 `通海港原始DOM优先重建路线.md`。

后续实现应优先围绕原始 `data/生产数据/无人机数据/正射/dom.tif` 的 ROI tile 推理、原始 DOM 分布下的模型微调、地图坐标钢轨候选、主线优先拓扑重建来展开。

## 当前结论

这轮不应继续把 Hermes 之前生成的 `final_centerline_network.shp` 当作最终答案去微调。它仍然有参考价值，但道岔区域差，根因不是单纯的 snap 距离或去重阈值，而是缺少“通海港站场拓扑”的上层约束。

以下是本文件历史阶段形成的双模型/拓扑骨架路线，现在只作为辅助参考，不再代表最新主干：

```text
单根铁轨语义模型
  -> 普通轨道中心线候选
  -> 最多三条平行股道分层
  -> 先生成连续贯通主线
  -> 再保留侧线/终端线
  -> 道岔区域模型只负责限定允许生成连接线的位置
  -> 后续在工作区内生成与主线/侧线相切的连接线
```

也就是说，最终中心线不再是一堆平等线段互相吸附，而应是：

- 一条近似直线、贯穿全场的 `main_through_track`。
- 一到两条局部平行侧线、装卸线或停留线，允许真实死端。
- 若干道岔/渡线连接边，只能在道岔工作区内生成，并且要与被连接股道方向连续或近似相切。

轻量转换检查后，这批点云转换到 `EPSG:32651` 后大致覆盖：

```text
X: 315438 - 315709
Y: 3521910 - 3522347
```

这个范围落在原始 DOM 和对齐走廊 DOM 的轨道区域内，因此可以作为中心线重建的重要几何参考。

需要区分两种“间距”：

- `轨距`：同一股轨道左右两根钢轨之间的距离，用来判断语义分割中两条 rail mask 是否属于同一股轨道。
- `股道间距 / 线间距`：相邻两股轨道中心线之间的距离，用来约束最多三条平行股道的 band 分层。

推荐用法：

1. 先把手持 LAS 统一转换或按需重投影到 `EPSG:32651`。
2. 在点云中提取钢轨头部的两条线性高密脊线，估计实际轨距和局部轨向。
3. 将 DOM 语义分割得到的钢轨候选像素转成地图坐标。
4. 只允许距离接近实测轨距的两根钢轨被配对成同一股轨道。
5. 用点云估计的股道间距约束 band 分层，避免把相邻股道的钢轨误配成一股轨道。
6. 后续中心线生成后，再用点云查询 `Z`，并检查左右钢轨高度连续性。

这条路线比单纯调 `min-pair-gap / max-pair-gap` 更稳。像素间距会随切片、旋转、重采样而变化，但点云中的实测轨距和股道间距是米制几何约束，可以回到地图坐标里统一判断。

### 道岔区污染与排除策略

手持点云覆盖了不少道岔区。道岔区里支线以弧线并入主线，还可能出现护轨、尖轨、交叉构件、设备和扫描段重叠；如果把这些点和普通直线段一起统计，全局横向剖面会被污染。因此轨距标定和股道间距估计必须区分两类点：

- `calibration points`：普通直线/平行股道附近的点，用于估计同一股轨道左右钢轨间距。
- `switch-zone points`：道岔工作区内的点，不参与全局轨距标定，保留给后续局部道岔连接、切向关系和轨头高度验证。

```text

```

正式生产流程里，排除区来源应换成原始 DOM 上推理出的 `switch_area` polygon。具体不是把 DOM 坐标“反向塞回手扫点云原坐标”，而是：

```text
原始 DOM tile 预测 switch_area mask
  -> 按 tile georef 转成 EPSG:32651 polygon
  -> 手扫 LAS 逐块转换到 EPSG:32651
  -> 用 polygon/buffer 判断点是否落入道岔工作区
  -> 标定轨距时排除这些点
  -> 道岔局部几何分析时单独使用这些点
```

当前要保守一点：现有道岔模型主要在旋转裁切后的走廊 DOM 上验证过，直接跑原始大 DOM 会有分布迁移风险。下一阶段应先在原始 DOM tile 上做小范围推理和 DOM 叠图 QA，确认 `switch_area` polygon 可信后，再把它作为正式 LAS 排除掩膜。

注意事项：

- 手持点云只覆盖部分轨道，不能当作全场唯一真值。
- 点云中会包含枕木、道砟、设备、遮挡和扫描噪声，需要先提取钢轨头或线性脊线，不能直接把所有点都拿来拟合中心线。
- 对没有手持点云覆盖的区段，仍应以 DOM 语义分割 + 站场拓扑先验为主。
- 当前最合适的定位是：手持点云用于校准轨距、校验股道间距、补 `Z`、验证局部道岔几何，而不是完全替代视觉模型。

下一步融合建议：

1. 修改中心线候选提取逻辑，除了像素 `gap_px` 外，同时计算左右钢轨候选的地图距离 `gap_m`。
2. 使用 `target_gauge_m = 1.50`、`gauge_accept_range_m = 1.35 - 1.65` 过滤左右钢轨配对。
3. 对落在手持 LAS 覆盖范围内的候选，提高点云验证通过的候选权重。
4. 对落在点云覆盖范围外的候选，继续使用 DOM 分割概率、连续性和站场拓扑先验。
5. 后续生成中心线后，用 LAS 点云给中心线补局部 `Z`，并检查左右钢轨高度连续性。

## 当前不能误解的地方

- `topology_skeleton.geojson` 不是最终交付中心线，它是“主线优先”的第一版骨架。
- 当前 v7 训练集来自旋转裁切后的轨道走廊图，不是原始 `data/生产数据/无人机数据/正射/dom.tif` 的直接切片。
- `switch_model_workzones.geojson` 是允许生成道岔连接的区域，不是道岔中心线本身。
- 当前 CVAT 没有 `switch_center` 点，所以不能基于岔心点精确构造相切位置。
- 侧线有死端是合理现象，不能为了降低 dead-end 数量把所有端点强行接回主线。
- 后续如果发现主线 band 选错，应优先检查 DOM 叠加和手绘参考中心线，而不是盲目调 snap tolerance。
- 点云单轨补全只在已有中心线连续性或锚线约束下启用，不能拿来在道岔区无条件造连接线。
