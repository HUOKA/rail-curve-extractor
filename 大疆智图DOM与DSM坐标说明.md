# 大疆智图 DOM 与 DSM 坐标信息说明

## 结论

大疆智图输出的二维正射图 `dom.tif` 本身主要存储影像颜色信息，即每个像素的 `R/G/B/A` 值；它不是给每个像素单独存储一组 `X/Y/Z` 坐标。

但是，`dom.tif` 是带空间参考信息的 GeoTIFF。它保存了坐标系、左上角位置、像元大小等地理配准参数。因此，DASView 或其他 GIS 软件打开后，可以根据这些参数计算任意像素对应的二维平面坐标 `X/Y`。

如果需要获得 `Z` 高程值，则通常需要结合 `dsm.tif` 或点云文件 `LAS/PNTS` 查询或插值。`Z` 不来自 `dom.tif` 的 RGB 像素本身。

## DOM 正射图存储的信息

本项目的 `dom.tif` 元数据信息如下：

```text
文件类型：BigTIFF / GeoTIFF
尺寸：36559 x 111549 像素
波段数：4 个波段
数据类型：8-bit unsigned integer
像元大小：0.0326 m/pixel
坐标系：WGS 84 / UTM zone 51N
EPSG：32651
NoData：0
```

其中 4 个波段对应：

```text
R：红色
G：绿色
B：蓝色
A：透明度或有效区域掩膜
```

因此，`dom.tif` 的像素值主要是颜色值，不是高程值，也不是完整三维坐标。

## DOM 像素坐标如何换算为地图坐标

`dom.tfw` 中记录了如下仿射变换参数：

```text
0.0326000000
0.0000000000
0.0000000000
-0.0326000000
314743.8353582112
3522560.9323315979
```

含义可以理解为：

```text
每个像素宽度：0.0326 m
每个像素高度：0.0326 m
左上角像素中心 X：314743.8353582112
左上角像素中心 Y：3522560.9323315979
```

设某个像素的列号为 `col`，行号为 `row`，且从 0 开始计数，则该像素中心点的二维地图坐标可按以下公式计算：

```text
X = 314743.8353582112 + col * 0.0326
Y = 3522560.9323315979 - row * 0.0326
```

其中，`Y` 使用减号，是因为影像坐标中行号向下增加，而地图坐标中北方向 `Y` 向上增加。

所以，DASView 打开 `dom.tif` 后，不需要每个像素都预先存一份坐标。它只要读取 GeoTIFF/TFW 中的空间参考参数，就可以实时计算出任意像素的 `X/Y` 平面位置。

## DSM 存储的信息

本项目的 `dsm.tif` 是数字表面模型，即 Digital Surface Model。它与 DOM 不同，DSM 的每个像素存储的是一个高程值 `Z`。

`dsm.tif` 元数据信息如下：

```text
文件类型：BigTIFF / GeoTIFF
尺寸：18252 x 55689 像素
波段数：1 个波段
数据类型：32-bit float
像元大小：0.0653 m/pixel
坐标系：WGS 84 / UTM zone 51N
EPSG：32651
NoData：-9999
压缩方式：Deflate
瓦片大小：256 x 256 像素
```

这里的 1 个波段就是高程波段。也就是说：

```text
DSM 像素值 = 该位置的表面高程 Z
```

本次抽样读取到的有效 DSM 高程值大致为：

```text
最小值约：12.06 m
中位数约：13.49 m
最大值约：16.80 m
```

以上只是抽样结果，不代表全图严格统计值，但可以说明 `dsm.tif` 中确实存储的是米级高程数据。

## DSM 像素坐标如何换算为 X/Y/Z

`dsm.tfw` 中记录了如下参数：

```text
0.0653000000
0.0000000000
0.0000000000
-0.0653000000
314743.8350626038
3522560.9320409116
```

设 DSM 中某个像素的列号为 `col`，行号为 `row`，则：

```text
X = 314743.8350626038 + col * 0.0653
Y = 3522560.9320409116 - row * 0.0653
Z = dsm.tif 中该像素的浮点数值
```

因此，DSM 可以提供 `Z`，但它同样不是逐像素单独存 `X/Y`。`X/Y` 是由空间参考参数计算出来的，`Z` 是栅格像素值本身。

## DOM 与 DSM 的关系

可以简单理解为：

```text
DOM：看颜色，看影像纹理
DSM：看高度，看表面高程
```

两者的空间参考一致，都是：

```text
WGS 84 / UTM zone 51N
EPSG:32651
```

但分辨率不同：

```text
DOM：0.0326 m/pixel
DSM：0.0653 m/pixel
```

也就是说，DSM 的分辨率约为 DOM 的一半。查询 DOM 某个像素的三维坐标时，通常会先由 DOM 算出 `X/Y`，再到 DSM 或点云中查询对应位置的 `Z`。

常见流程如下：

```text
DOM 像素 row/col
        ↓
根据 DOM 的仿射变换计算 X/Y
        ↓
使用 X/Y 到 DSM 或点云中查询/插值
        ↓
得到 Z
        ↓
形成 X/Y/Z
```

## 瓦片大小是什么意思

`dsm.tif` 内部显示瓦片大小为：

```text
256 x 256 像素
```

这表示这个很大的 DSM 文件在内部不是按整张图一次性存储，而是被拆成许多 `256 x 256` 的小块进行组织和压缩。

这样做的作用是提高读取和显示效率。例如软件只需要查看某一小块区域时，只读取对应的几个瓦片即可，不需要把整张大图全部加载。

对于本项目 DSM：

```text
DSM 像元大小：0.0653 m/pixel
瓦片大小：256 x 256 像素
```

因此，一个 DSM 瓦片覆盖的实际地面范围约为：

```text
256 * 0.0653 = 16.7168 m
```

即约：

```text
16.7 m x 16.7 m
```

需要注意，瓦片大小不是精度，也不是坐标单位。它只是文件内部组织数据的方式。

## 关于 DASView 的坐标解算

相关人员提到 DASView 导入正射图后会自动构建三维金字塔，并解算图片中每个像素对应的坐标。结合本项目文件结构，可以理解为：

1. 对 DOM 影像，DASView 可以利用 GeoTIFF/TFW 中的坐标系、左上角坐标和像元大小，计算像素对应的 `X/Y`。
2. 对 `Z` 值，DASView 通常还需要读取 DSM 或点云数据。
3. 三维金字塔更可能是为了加快三维点云或大规模栅格数据的显示、缩放、定位和拾取。

本项目中已经存在类似三维瓦片结构：

```text
D:\正射\lidars\terra_pnts\tileset.json
D:\正射\lidars\terra_pnts\Block*\*.pnts
```

这类文件用于三维点云的分块、层级化显示和快速加载。

因此，如果 DASView 中点击 DOM 上某一点能够显示 `X/Y/Z`，比较合理的解释是：

```text
X/Y：由 DOM 的地理配准参数计算得到
Z：由 DSM 或点云数据查询/插值得到
```

而不是：

```text
DOM 的每个 RGB 像素里直接存了 X/Y/Z
```

## 开源实现与参考

DASView 自身的具体实现是否开源，本次未发现可确认的官方开源源码。因此不能直接说 DASView 的内部算法是开源的。

但它所涉及的基础方法是通用 GIS/三维瓦片技术，相关开源工具和思路包括：

| 类型 | 常见开源实现 |
| --- | --- |
| GeoTIFF 坐标换算 | GDAL、rasterio |
| 点云处理 | PDAL |
| 点云三维瓦片/金字塔 | Potree、Entwine |
| 三维瓦片显示 | Cesium 3D Tiles |

其中，像素坐标到地图坐标的核心换算属于标准 GeoTIFF 仿射变换；三维点云金字塔则属于大规模点云数据的分块、分层索引与可视化加载技术。
